Notes:

- `teardown` lists literal snake_case table names in FK-safe drop order.
  Gazette gets this right; contact-form's PascalCase entries are wrong - do
  not copy contact-form here (see FIELD_NOTES Gazette "Known Limitations").
- `puckBlocks` is empty **on purpose**. The six Boards body blocks (section 7)
  are thread-opener palette blocks, module-internal like Gazette's body
  palette, never placed on core pages, therefore never manifest-registered.
- `publicBasePath: "boards"` is fixed. Changing the public URL prefix means
  editing the manifest and redeploying. There is deliberately no runtime admin
  setting for it.
- The `extensionPoints` entry feeds the core dashboard widget (section 5.10).
  The generic extension-point mechanism already exists
  (`lib/modules/extension-points.ts`, generated by
  `scripts/generate-module-extension-points.mjs`); only the dashboard's
  consumption of the `core.admin-dashboard-widgets` point is new core work
  (see section 5.10 - small, intentional, not a protected category).

Versioning: patch bumps in both `package.json` and `cactus.module.json` per
release; the GitHub release tag tracks the **manifest** version (contact-form
precedent). Module is cloned at build by `scripts/checkout-modules.mjs`; the
`modules.json` entry (`{ "name": "boards", "repoUrl": "https://github.com/cactus-foundation-modules/boards", "version": "0.1.0" }`)
is added only once the repo exists - an implementation-time step, not part of
this spec's diff.

## 3. Database schema

Single file `modules/boards/migrations/001_initial.sql`, applied by
`scripts/run-module-migrations.mjs`, tracked in the core `ModuleMigration`
table. All DDL idempotent (`CREATE TABLE IF NOT EXISTS`,
`CREATE INDEX IF NOT EXISTS`). Conventions copied from Gazette's
`001_initial.sql`:

- `id TEXT NOT NULL DEFAULT gen_random_uuid()::text` primary keys (DB-side
  default, **not** app-side UUIDs).
- `TIMESTAMP(3)` with `DEFAULT CURRENT_TIMESTAMP` for `created_at`/`updated_at`.
- Status columns are `TEXT` with `CHECK` constraints, not enums.
- FKs to core users reference `"User" ("id")` with the ON DELETE behaviour
  stated per table below. GDPR consequences in section 5.11.
- Media references store `Media.id` as plain `TEXT` with **no FK** (mirrors
  `gz_posts.featured_image_id` / `InfoPage.ogImageId`; render falls back
  gracefully).

Twenty-nine tables. Columns marked `?` are nullable.

### Structure

**`brd_categories`** - moderator-defined labels that group boards visually
on the index.
`id`, `title`, `position` INTEGER NOT NULL DEFAULT 0, `created_at`,
`updated_at`.
Index: `(position)`.

**`brd_boards`** - top-level boards.
`id`, `category_id?` FK → brd_categories (SET NULL; uncategorised boards
render in a trailing unlabelled group),
`title`, `slug` (unique), `description?`, `position` INTEGER NOT NULL DEFAULT 0,
`icon_emoji?` TEXT, `icon_media_id?` TEXT (Media.id, no FK; the structure
screen lets an admin set one or the other, emoji taking precedence if both
somehow exist),
`is_locked` BOOLEAN NOT NULL DEFAULT false (no new threads),
`visibility` TEXT NOT NULL DEFAULT 'PUBLIC' CHECK IN
('PUBLIC','MEMBERS','PRIVATE') - PUBLIC readable by anyone; MEMBERS requires
a session; PRIVATE visible and readable only to that board's moderators,
Global Moderators and admins, and excluded from listings, search, sitemap
and feed for everyone else. Sub-boards inherit their parent board's
visibility (no column of their own),
`noindex` BOOLEAN NOT NULL DEFAULT false (per-board search-engine opt-out,
independent of visibility: a PUBLIC board with `noindex` stays readable by
anyone but its threads emit a `noindex` meta tag and are excluded from the
sitemap and RSS feed),
`min_post_length?` INTEGER (minimum characters for thread bodies and replies
in this board; NULL = no minimum),
`word_filter?` JSONB (array of lowercase terms; NULL = no filter; matched
against new thread/post text at submission - hits go to the moderation
queue, see 5.3), `created_at`, `updated_at`.
Index: `(category_id, position)`.

**`brd_sub_boards`** - one level of nesting only (enforced by schema shape;
sub-boards cannot have children).
`id`, `board_id` FK → brd_boards (CASCADE), `title`, `slug`, `description?`,
`position` INTEGER NOT NULL DEFAULT 0, `is_locked` default false,
`created_at`, `updated_at`.
Unique `(board_id, slug)`. Index: `(board_id, position)`.

**`brd_tags`** - `id`, `name`, `slug` (unique), `created_at`.

### Content

**`brd_threads`**
`id`, `board_id` FK → brd_boards (CASCADE),
`sub_board_id?` FK → brd_sub_boards (CASCADE),
`title`, `slug` (unique, globally - drives `/boards/t/<slug>`),
`author_id?` FK → "User" (SET NULL), `author_name` TEXT NOT NULL (display
snapshot at creation time; survives account deletion, see 5.11),
`opener_data?` JSONB (Puck Data from the thread-opener body palette),
`status` TEXT NOT NULL DEFAULT 'PUBLISHED' CHECK IN
('PUBLISHED','PENDING','HIDDEN','DELETED','ARCHIVED') - ARCHIVED threads
stay listed and readable but accept no new replies, reactions or poll votes
(read-only end state, distinct from the reversible `is_locked` flag on a
live thread),
`is_pinned` default false, `is_locked` default false,
`is_global_announcement` BOOLEAN NOT NULL DEFAULT false (renders pinned at
the top of the `/boards` index and every board/sub-board listing regardless
of which board it lives in; set/unset is a Global Moderator action, logged),
`view_count` INTEGER NOT NULL DEFAULT 0,
`reply_count` INTEGER NOT NULL DEFAULT 0 (denormalised, maintained on
post create/delete),
`last_post_at?` TIMESTAMP(3) (denormalised, drives board listing sort;
set on **post creation only** - editing a post never updates it, so edits
do not bump a thread, and deletions leave it unchanged),
`ip_address?` TEXT (submitter IP, for IP-ban moderation; mirrors
`gz_comments.ip_address`),
`imported_from?` TEXT (source id for imports, e.g. `phpbb:1234`),
`created_at`, `updated_at`.
Indexes: `(board_id, is_pinned DESC, last_post_at DESC)`,
`(sub_board_id, last_post_at DESC)`, `(author_id)`, `(status, created_at DESC)`,
`(imported_from)`, partial on `(is_global_announcement)` WHERE true.

**`brd_posts`** - replies (the opener body lives on the thread row).
`id`, `thread_id` FK → brd_threads (CASCADE),
`author_id?` FK → "User" (SET NULL), `author_name` TEXT NOT NULL (snapshot),
`body_html` TEXT NOT NULL (sanitised rendered output),
`body_source` JSONB (editor document, source of truth for edits),
`reply_to_post_id?` FK → brd_posts (SET NULL; quoted-reply pointer, flat
rendering - threads are linear, not nested),
`status` TEXT NOT NULL DEFAULT 'PUBLISHED' CHECK IN
('PUBLISHED','PENDING','HIDDEN','DELETED'),
`edited_at?`, `edited_by?` FK → "User" (SET NULL),
`ip_address?` TEXT (submitter IP, for IP-ban moderation),
`imported_from?` TEXT, `created_at`, `updated_at`.
Indexes: `(thread_id, created_at)`, `(author_id)`, `(status, created_at DESC)`,
`(imported_from)`.

**`brd_post_revisions`** - edit history, moderator-visible.
`id`, `post_id` FK → brd_posts (CASCADE), `body_html`, `body_source` JSONB,
`edited_by?` FK → "User" (SET NULL), `created_at`.
Index: `(post_id, created_at DESC)`.

**`brd_thread_tags`** - `(thread_id, tag_id)` composite PK, CASCADE both ways.
Index: `(tag_id)`.

**`brd_drafts`** - composer auto-save (5.3). One reply draft per user per
thread, plus any number of new-thread drafts keyed by board.
`id`, `user_id` FK → "User" (CASCADE),
`thread_id?` FK → brd_threads (CASCADE) - set for reply drafts, NULL for
new-thread drafts,
`board_id?` FK → brd_boards (CASCADE) - context for new-thread drafts,
`title?`, `opener_data?` JSONB (Puck Data, new-thread drafts),
`body_source?` JSONB (composer document, reply drafts),
`created_at`, `updated_at`.
Partial unique index on `(user_id, thread_id)` WHERE `thread_id IS NOT NULL`.
Index: `(user_id, updated_at DESC)`.

### Engagement

**`brd_post_reactions`** - registered users only (deliberate divergence from
Gazette's anonymous visitor-token reactions; Boards posting is
registered-only, so reactions key on the user).
`id`, `post_id` FK → brd_posts (CASCADE), `user_id` FK → "User" (CASCADE),
`emoji` TEXT NOT NULL, `created_at`.
Unique `(post_id, user_id, emoji)`. Index: `(post_id)`.
The UI reuses the Gazette `Reactions.tsx` component pattern
(`modules/gazette/components/public/Reactions.tsx`), swapping visitor token
for session identity.

**`brd_thread_views`** - anonymous view dedupe, exact mirror of
`gz_post_views` + `modules/gazette/lib/visitor.ts` (cookie renamed
`cactus-boards-vid`).
`id`, `thread_id` FK → brd_threads (CASCADE), `visitor_token`, `viewed_at`.
Unique `(thread_id, visitor_token)`.

**`brd_bookmarks`** - user-saved threads (5.14).
`id`, `thread_id` FK → brd_threads (CASCADE), `user_id` FK → "User" (CASCADE),
`created_at`.
Unique `(thread_id, user_id)`. Index: `(user_id, created_at DESC)`.

**`brd_read_state`** - per-user per-thread read tracking (5.14).
`user_id` FK → "User" (CASCADE), `thread_id` FK → brd_threads (CASCADE),
`last_read_post_at` TIMESTAMP(3) NOT NULL, `updated_at`.
Composite PK `(user_id, thread_id)`. Index: `(user_id)`.
A thread is unread for a user when it has no row or
`last_post_at > last_read_post_at`; board listings derive their unread
counts from this join (members only; anonymous visitors see no unread state).

**`brd_polls`** - one optional poll per thread.
`id`, `thread_id` FK → brd_threads (CASCADE, unique), `question`,
`allow_multiple` BOOLEAN NOT NULL DEFAULT false, `closes_at?`,
`created_at`.

**`brd_poll_options`** - `id`, `poll_id` FK → brd_polls (CASCADE),
`label`, `position` INTEGER NOT NULL DEFAULT 0.

**`brd_poll_votes`** - `id`, `option_id` FK → brd_poll_options (CASCADE),
`poll_id` FK → brd_polls (CASCADE), `user_id` FK → "User" (CASCADE),
`created_at`. Unique `(option_id, user_id)`. Index `(poll_id, user_id)` (used
to enforce single-choice when `allow_multiple` is false, app-side).

### People

**`brd_user_profiles`** - forum identity, deleted with the user (personal
data; mirrors `gz_author_profiles`).
`id`, `user_id` FK → "User" (CASCADE, unique),
`username` TEXT NOT NULL (unique; public handle, drives `/boards/u/<username>`),
`signature?` TEXT, `avatar_id?` (Media.id, no FK), `bio?` TEXT,
`post_count` INTEGER NOT NULL DEFAULT 0 (denormalised),
`last_seen_at?`, `created_at`, `updated_at`.

**`brd_moderator_assignments`** - board-scoped moderators and the Global
Moderator flag in one table (module-side, like `gz_user_roles`; **no core
Role/Permission table changes**).
`id`, `user_id` FK → "User" (CASCADE),
`board_id?` FK → brd_boards (CASCADE) - **NULL means Global Moderator**,
`assigned_by?` TEXT, `created_at`.
Unique `(user_id, board_id)` plus a partial unique index on `(user_id)` WHERE
`board_id IS NULL` (one global flag per user).
Assignment is core-admin only (`isAdmin`), same rule and same
screen-hiding behaviour as Gazette's roles screen.

**`brd_bans`** - forum-level sanctions (does not touch the core user's
account or sessions).
`id`, `user_id` FK → "User" (CASCADE), `reason?`,
`expires_at?` (NULL = indefinite), `created_by?` TEXT, `created_at`.
Index: `(user_id, expires_at)`.

**`brd_ip_bans`** - IP-level blocks, checked at thread/post submission
alongside the user-level ban check (source IPs come from the `ip_address`
columns on threads/posts; the mod UI offers "ban this IP" from an offending
item).
`id`, `ip_address` TEXT NOT NULL, `reason?`,
`expires_at?` (NULL = indefinite), `created_by?` TEXT, `created_at`.
Unique `(ip_address)`. Index: `(ip_address, expires_at)`.

### Moderation

**`brd_moderation_queue`** - items awaiting review (pre-moderated first
posts, filter hits).
`id`, `item_type` TEXT NOT NULL CHECK IN ('THREAD','POST'),
`item_id` TEXT NOT NULL, `reason` TEXT NOT NULL,
`status` TEXT NOT NULL DEFAULT 'OPEN' CHECK IN ('OPEN','APPROVED','REJECTED'),
`resolved_by?` FK → "User" (SET NULL), `resolved_at?`, `created_at`.
Index: `(status, created_at)`.
No FK on `item_id` (polymorphic); app code joins on `item_type`.

**`brd_reports`** - member-filed flags.
`id`, `item_type` CHECK IN ('THREAD','POST'), `item_id`,
`reporter_id?` FK → "User" (SET NULL), `reason` TEXT NOT NULL,
`status` TEXT NOT NULL DEFAULT 'OPEN' CHECK IN ('OPEN','RESOLVED','DISMISSED'),
`resolved_by?` FK → "User" (SET NULL), `resolved_at?`, `created_at`.
Index: `(status, created_at)`.

**`brd_moderation_log`** - append-only audit of moderator actions
(hide/delete/lock/pin/archive/move/split/announce/warn/ban/ip-ban/approve/reject).
`id`, `actor_id?` FK → "User" (SET NULL), `actor_name` TEXT NOT NULL
(snapshot), `action` TEXT NOT NULL, `item_type?`, `item_id?`,
`detail?` JSONB, `created_at`.
Index: `(created_at DESC)`.

### Subscriptions and notifications (consumer side)

**`brd_thread_subscriptions`** - `id`, `thread_id` FK → brd_threads (CASCADE),
`user_id` FK → "User" (CASCADE), `created_at`.
Unique `(thread_id, user_id)`. Index `(user_id)`.

**`brd_board_subscriptions`** - same shape against `brd_boards`.
Unique `(board_id, user_id)`. Index `(user_id)`.

**`brd_notification_prefs`** - per-user Boards delivery preferences.
`user_id` PK FK → "User" (CASCADE),
`mode` TEXT NOT NULL DEFAULT 'IMMEDIATE' CHECK IN ('IMMEDIATE','DIGEST','OFF'),
`email_enabled` BOOLEAN NOT NULL DEFAULT true,
`last_digest_at?`, `updated_at`.
The digest cron uses `last_digest_at` as its low-water mark; no separate
queue table.

### Everything else

**`brd_thread_templates`** - mirrors `gz_post_templates`.
`id`, `title`, `builder_data?` JSONB (Puck Data), `created_at`, `updated_at`.

**`brd_import_jobs`** - one row per import run.
`id`, `source` TEXT NOT NULL CHECK IN ('PHPBB','DISCOURSE'),
`status` TEXT NOT NULL DEFAULT 'PENDING' CHECK IN
('PENDING','RUNNING','DONE','FAILED'),
`stats?` JSONB (counts: boards/threads/posts imported, skipped, errors),
`error?` TEXT, `created_by?` TEXT, `created_at`, `updated_at`.

**`brd_settings`** - singleton (`id = 'singleton'`, CHECK-enforced, seeded by
the migration with `INSERT ... ON CONFLICT DO NOTHING`, exactly like
`gz_settings`). Columns:
`threads_per_page` INTEGER NOT NULL DEFAULT 20,
`posts_per_page` INTEGER NOT NULL DEFAULT 20,
`rss_enabled` BOOLEAN NOT NULL DEFAULT true,
`feed_title?`, `feed_description?`,
`reactions_enabled` BOOLEAN NOT NULL DEFAULT true,
`reaction_set?` JSONB (NULL = code default, mirrors Gazette),
`signatures_enabled` BOOLEAN NOT NULL DEFAULT true,
`signature_max_length` INTEGER NOT NULL DEFAULT 500 (characters, enforced on
profile save),
`min_account_age_days` INTEGER NOT NULL DEFAULT 0 (accounts younger than
this cannot create threads or posts at all - hard 403 with a friendly
message; 0 = off; independent of the approval queue),
`first_post_count` INTEGER NOT NULL DEFAULT 3 (see queue rule below; 0 = off),
`first_post_account_age_days` INTEGER NOT NULL DEFAULT 7,
`post_cooldown_seconds` INTEGER NOT NULL DEFAULT 30,
`posts_per_hour_limit` INTEGER NOT NULL DEFAULT 20,
`edit_window_minutes` INTEGER NOT NULL DEFAULT 0 (authors may edit their own
posts only within this many minutes of posting; 0 = unlimited; moderators
and admins are exempt),
`show_view_counts` BOOLEAN NOT NULL DEFAULT true,
`updated_at`.

Queue rule (new-account moderation): a thread/post is created `PENDING` with
a `brd_moderation_queue` row when the author's `post_count <
first_post_count` **and** their core account is younger than
`first_post_account_age_days` days. Established accounts skip the queue even
on their first Boards post.

### Schema flag list (auth/permission-adjacent columns)

Reviewed against the protected categories:

- `brd_moderator_assignments`, `brd_bans`, `brd_ip_bans` - module-side
  authorisation state only. No core `Role`, `Permission`, session, or
  Prisma-migration change is required or permitted. Same architecture as
  `gz_user_roles`.
- No table stores credentials, tokens, or session material. The only
  personal data beyond core `User` is `brd_user_profiles` (CASCADE-deleted
  with the user) and the `author_name` snapshots (see GDPR, 5.11).
- Zero changes to `prisma/schema.prisma` and zero edits to the core init
  migration.

## 4. Access model summary

Three layers, all existing mechanisms:

1. **Core permissions** (manifest-declared, appear in the `/cactus-admin/roles`
   matrix automatically via the module registration path):
   - `boards.access` - gates the admin nav entry and admin screens' baseline.
   - `boards.manage` - gates structural admin (boards/sub-boards CRUD,
     settings, imports, templates, tags).
2. **Module roles** (`brd_moderator_assignments`): board moderator (row with
   `board_id`) and Global Moderator (`board_id IS NULL`). Checked by
   `modules/boards/lib/permissions.ts`, modelled on
   `modules/gazette/lib/permissions.ts` (`getBoardsAccess(user)` returning a
   capabilities object; core `isAdmin(user)` passes everything).
3. **Membership**: any logged-in core user
   (`getSessionFromCookie()` from `lib/auth/session.ts`) may read
   MEMBERS-visibility boards, create threads/posts/reactions/votes, subscribe, and
   file reports - unless banned (`brd_bans` active row) or the target is
   locked. Anonymous visitors may read public boards and increment view
   counts; they can do nothing else.

Capability matrix (module-side, enforced in `lib/permissions.ts`):

| Action | Member | Board Moderator (that board) | Global Moderator | Core Admin |
|---|---|---|---|---|
| Create thread/post | yes (unless banned/locked) | yes | yes | yes |
| Edit own post (within `edit_window_minutes`; revisions logged) | yes | exempt from window | exempt | exempt |
| Edit/hide/delete any post | - | in board | yes | yes |
| Lock/pin/archive/move/split threads | - | in board | yes | yes |
| Approve queue / resolve reports | - | in board | yes | yes |
| Ban members / IP bans / global announcements | - | - | yes | yes |
| Manage boards/settings/imports | - | - | - | needs `boards.manage` |
| Assign moderators | - | - | - | `isAdmin` only |

## 5. Features

### 5.1 Boards and sub-boards

Admin CRUD under `/cactus-admin/m/boards/structure` (gated `boards.manage`):
create/rename/describe/reorder (position drag, mirrors the Gazette series
reorder pattern), category management (create/rename/reorder/delete labels;
deleting a category leaves its boards uncategorised via SET NULL), board
icon (emoji picker or media upload), lock, visibility select
(Public / Members / Private, see section 3), hide-from-search-engines
toggle (`noindex`), minimum post length, per-board
word filter (textarea, one term per line, stored lowercased in
`brd_boards.word_filter`), delete (CASCADE warning showing thread/post
counts first). One level of nesting only. Public listing at
`/boards` shows boards with sub-boards inline, thread/post counts and last
activity (from denormalised `last_post_at`), grouped under category headings
(`brd_categories`, position order) with uncategorised boards in a trailing
unlabelled group; board icons (emoji or media image) render beside titles
where set. PRIVATE boards are omitted entirely for viewers without moderator
capability on them.

### 5.2 Threads

Created by any member from a board or sub-board page. Title + opener composed
in the thread-opener Puck editor (section 7), optional tags, optional poll.
Slug generated from title, globally unique, deduplicated with a numeric
suffix. Pin, lock and archive are three distinct moderator actions: pin
floats the thread in its board, lock reversibly stops new replies, archive
puts it in the read-only end state described in section 3 (no replies,
reactions or poll votes; still listed and readable). Global announcements: a Global
Moderator (or admin) can flag a thread `is_global_announcement`; it then
renders in a pinned "Announcements" strip at the top of the `/boards` index
and every board/sub-board listing, above board-local pins. Setting and
clearing the flag are logged moderation actions. Moving a thread updates
`board_id`/`sub_board_id` and logs to `brd_moderation_log`.

Thread splitting (moderator action): from any reply's mod menu, "split
thread from here" - the moderator supplies a new title (and optionally a
different board); the selected post and every later post in the thread move
to a new thread with `opener_data` NULL (the moved posts simply start the
thread; `brd_threads.opener_data` is already nullable for exactly this
case). Both threads get their `reply_count`/`last_post_at` recomputed
transactionally, both get `brd_moderation_log` rows, and subscriptions stay
on the original thread (subscribers are not silently moved).

Thread view: posts paginate oldest-first by default with a member-facing
sort toggle to newest-first; the choice is reflected in the querystring
(`?sort=newest`) so links share the view, and persisted client-side
(localStorage) - no schema involvement. View counting via
the visitor-token dedupe described in section 3. `reply_count` and
`last_post_at` maintained transactionally with post creation (see the
`last_post_at` rule in section 3: creation only, never edits).

### 5.3 Posts (replies)

Rich-text composer (section 7 note on the editor), quoted replies via
`reply_to_post_id` (renders a collapsed quote block linking to the source
post; threads stay linear). Edits write the previous body to
`brd_post_revisions` and set `edited_at`/`edited_by`; an "edited" marker
renders publicly, revision diffs are moderator-only. Author edits are only
accepted within `brd_settings.edit_window_minutes` of the post's creation
(0 = unlimited; 403 with a friendly message after that); moderator edits
ignore the window. Deletion by the author
soft-deletes (`status = 'DELETED'`, body replaced in render with "Post
removed"); hard delete is moderator-only.

Submission gauntlet, in order (thread creation and replies alike):

1. Session + not banned (`brd_bans`) + IP not banned (`brd_ip_bans`, active
   rows only) + account age ≥ `min_account_age_days`.
2. Target board/sub-board/thread not locked; thread not archived.
3. Body meets the board's `min_post_length` (when set); zod validation
   otherwise.
4. Turnstile (`verifyTurnstile` from `lib/auth/turnstile.ts`, fail-open when
   unconfigured), exactly as
   `modules/contact-form/app/api/contact/submit/route.ts` does.
5. Rate limits from settings: `post_cooldown_seconds` between posts and
   `posts_per_hour_limit` per user (reuse the core rate-limit helper the
   Gazette public comments route uses, IP + user scoped, with the limits read
   from `brd_settings` rather than hardcoded).
6. Moderation gate - item lands `PENDING` + queue row when either applies:
   the new-account rule (`post_count < first_post_count` and account younger
   than `first_post_account_age_days`, see section 3) or a word-filter hit
   (any `brd_boards.word_filter` term present, case-insensitive, in the
   title or body text; queue `reason` records which). Otherwise `PUBLISHED`.

The submitter's IP is recorded on the row (`ip_address`) for later IP-ban
moderation.

Draft auto-save: the composer PUTs to the drafts endpoint on a debounce
(same 1.5s cadence as the core Puck editor autosave), storing into
`brd_drafts` - reply drafts keyed `(user, thread)`, new-thread drafts keyed
to the board. Returning to the composer offers recovery; successful
submission deletes the draft.

### 5.4 Identity and profiles

On first Boards interaction a `brd_user_profiles` row is created; username
defaults from the core display name (slugified, deduplicated) and is
user-editable once (then admin-only). Profile page `/boards/u/<username>`:
avatar (core media, `validateUpload`/`uploadMedia` from `lib/media/upload.ts`),
bio, post count, join date, recent threads/posts. Signature (when enabled in
settings) renders under each post, plain-formatting subset only, capped at
`brd_settings.signature_max_length` characters on save.

### 5.5 Roles and moderation

Moderator assignment screen `/cactus-admin/m/boards/moderators` - **core-admin
only, 404 (not an alert) for everyone else**, copying Gazette's Roles screen
behaviour verbatim. Moderation surfaces:

- Queue (`/cactus-admin/m/boards/moderation`): pending items with preview,
  approve/reject; reports list with resolve/dismiss.
- Inline moderator controls on public thread pages when the viewer has
  capability (hide/delete/lock/pin/archive/move, split-from-here on any
  reply; announce for Global Moderators).
- Warn user: a moderator action available from any post's mod menu and the
  member's profile mod menu. Takes a reason, writes a `brd_moderation_log`
  row (`action = 'warn'`, `item_type = 'USER'`, `item_id` = user id,
  `detail.reason`), and notifies the user (5.7). No separate table - a
  user's warning history and count are read from the log, and shown in the
  mod UI next to their name.
- Bulk moderation: the admin thread and post lists (and the queue) support
  multi-select with an action bar - hide, delete, lock, archive, move (to a
  picked board), approve, reject - applied in one request. Scope is checked
  per item; items the moderator cannot act on are skipped and reported back
  in the response (`{done, skipped}`). One log row per affected item, with
  `detail.bulk = true`. Pattern precedent: Gazette's `POST /posts/bulk` and
  `POST /comments/bulk`.
- Ban management (Global Moderator+): user bans and IP bans - list, add with
  optional expiry, lift; "ban this IP" shortcut from any post's mod menu
  (reads the row's `ip_address`).
- Log viewer (read-only table over `brd_moderation_log`).

Every moderation mutation writes a `brd_moderation_log` row.

### 5.6 Search

Module-side only, zero core migration. Postgres `ILIKE` over thread titles
plus a `to_tsvector('english', ...)` expression index option on
`brd_posts.body_html` **is not used** - instead the initial migration creates:

```sql
CREATE INDEX IF NOT EXISTS "brd_posts_body_search_idx"
  ON "brd_posts" USING GIN (to_tsvector('english', "body_html"));
CREATE INDEX IF NOT EXISTS "brd_threads_title_search_idx"
  ON "brd_threads" USING GIN (to_tsvector('english', "title"));
```

Public search endpoint queries with `plainto_tsquery`, filtered to
`status = 'PUBLISHED'` and boards visible to the requester (PRIVATE boards
only for their moderators/admins, MEMBERS only with a session), ranked by
`ts_rank` then recency.
Search UI on `/boards` (input in the index header) and results page
`/boards?q=...` (querystring, not a new route segment).

### 5.7 Notifications (consumer side only)

Boards **calls** a shared core notification interface; it does not implement
delivery, preferences UI beyond its own `mode` setting, or the bell. The core
side is a protected item (section 11) specified separately in
`NOTIFICATIONS_SPEC.md`. Until that ships, `modules/boards/lib/notify.ts`
wraps the interface behind a single function so Boards compiles and works
with a no-op/email-only fallback:

```ts
// modules/boards/lib/notify.ts
type BoardsNotification = {
  userId: string
  title: string        // "Alice replied to your thread"
  link: string         // site-relative, e.g. "/boards/t/some-thread#post-<id>"
  dedupeKey?: string
}
export async function notifyUser(n: BoardsNotification): Promise<void>
```

Triggers (all fire-and-forget, never block the write path):

- Reply in a subscribed thread → thread subscribers minus the poster.
- New thread in a subscribed board → board subscribers minus the author.
- Quoted reply (`reply_to_post_id`) → quoted post's author.
- @mention → each mentioned user. The submission handler parses
  `@username` tokens from the plain text of the opener/post body, matches
  them case-insensitively against `brd_user_profiles.username`, dedupes,
  drops the author's self-mentions, caps at 10 mentions per item (anti-spam),
  and fires `notifyUser` for each match. The renderer linkifies matched
  mentions to `/boards/u/<username>`; unmatched `@text` is left as plain
  text.
- Moderation outcome (approved/rejected from queue) → item author.
- Warning issued (5.5) → the warned user, with the reason.

Delivery respects `brd_notification_prefs.mode`: `IMMEDIATE` sends now (bell
via core interface when available + email via `sendEmail` from
`lib/email/index.ts` when `email_enabled`), `DIGEST` accumulates implicitly
(the cron recomputes from subscription activity since `last_digest_at`),
`OFF` suppresses everything.

Daily digest cron `POST /api/m/boards/cron/digest`, schedule `0 7 * * *`,
guarded by `Authorization: Bearer $CRON_SECRET` exactly like the contact-form
retention cron. It selects users with `mode = 'DIGEST'` and
`email_enabled = true`, gathers new activity in their subscriptions since
`last_digest_at`, sends one email per user via `sendEmail`, then advances
`last_digest_at`. No new env vars: `CRON_SECRET` already exists.

### 5.8 SEO and feeds

- `modules/boards/lib/sitemap.ts` exports `getPublicSitemapEntries(siteUrl)`
  (boards, sub-boards, published public threads), collected automatically by
  `collectModuleSitemapEntries()` in the generated router. Pattern:
  `modules/gazette/lib/sitemap.ts`.
- Global feed at `/boards/feed.xml` via `dispatchModulePublicRoute`: RSS 2.0,
  latest 20 published threads from public boards, title/description from
  `brd_settings`.
- Per-board feeds at `/boards/<board-slug>/feed.xml` and
  `/boards/<board-slug>/<sub-slug>/feed.xml`: same shape, scoped to the
  board/sub-board, titled from the board. Served only for PUBLIC,
  non-`noindex` boards; anything else 404s. All feeds honour
  `rss_enabled`. **Core note:** the existing core delegate covers only the
  one-segment `/[slug]/feed.xml` case (`app/(public)/[slug]/feed.xml/route.ts`,
  a literal segment because a route handler cannot share the `[...path]`
  page catch-all's folder). Nested feeds need two more generic delegates -
  `app/(public)/[slug]/[a]/feed.xml/route.ts` and
  `app/(public)/[slug]/[a]/[b]/feed.xml/route.ts` - each a two-line
  passthrough to `dispatchModulePublicRoute` with the extra path segments,
  containing no module names. This is the second intentional generic core
  change (alongside the dashboard extension point, 5.10); it touches no
  protected category. The module-side handlers live at
  `app/public/boards/[board]/feed.xml/route.ts` and
  `app/public/boards/[board]/[sub]/feed.xml/route.ts`.
- Thread pages emit title/description/OG metadata; non-PUBLIC board content
  (MEMBERS and PRIVATE alike) is `noindex` and excluded from sitemap/feed,
  as is any PUBLIC board with the per-board `noindex` toggle set.

moderation counts.

**Core summary widget** - ships with this work as the one intentional core

- Core change: `app/cactus-admin/page.tsx` (the dashboard) reads
  contributions for that point from the existing generated

- Boards contributes `boardsDashboardWidget`
  (`modules/boards/components/admin/DashboardWidget.tsx`): thread/post counts,

### 5.11 GDPR

Core account deletion (`DELETE /api/account`, `app/api/account/route.ts`)
deletes the `User` row. Boards handles this entirely module-side through FK
design - **no core hook, no core change**:

- `brd_user_profiles`, subscriptions, bookmarks, drafts, read state,
  reactions, poll votes, prefs, bans, moderator assignments:
  `ON DELETE CASCADE` (personal data, removed).
- `brd_threads.author_id`, `brd_posts.author_id`, and the various
  `resolved_by`/`edited_by` columns: `ON DELETE SET NULL`. Content survives.
- Rendering rule: when `author_id IS NULL`, display **"Deleted User"** and no
  profile link. The `author_name` snapshot column exists for imported content
  and for moderation-log legibility; public rendering of an authored item
  whose `author_id` has become NULL ignores the snapshot and shows
  "Deleted User" (the snapshot would otherwise keep personal data visible
  after erasure).
- `ip_address` on surviving threads/posts is retained (moderation evidence,
  `gz_comments.ip_address` precedent); once `author_id` is NULL it is no
  longer linked to an identifiable account. `brd_ip_bans` rows reference no
  user and persist.
- The core GDPR JSON export covers core data only; Boards adds
  `GET /api/m/boards/account/export` returning the member's profile, threads,
  posts, votes, subscriptions, bookmarks, drafts and read state as JSON,
  linked from their Boards profile settings.

### 5.12 Import wizard

`/cactus-admin/m/boards/import` (gated `boards.manage`), modelled on
`modules/gazette/lib/import/` + `ImportWizard.tsx` + the multipart gazette
import API:

- **phpBB**: XML export file. Maps forums → boards/sub-boards, topics →
  threads, posts → posts. BBCode converted to the composer's document format
  (supported subset; unknown tags degrade to plain text).
- **Discourse**: JSON export. Categories → boards, topics → threads,
  posts → posts.
- Authors: matched to existing core users by email where the export includes
  one; otherwise `author_id` stays NULL and `author_name` carries the
  imported handle (renders as the handle, since no erasure has occurred -
  the "Deleted User" rule in 5.11 applies only when an `author_id` was set
  and later nulled; implementation distinguishes via a
  `was_authored` inference: imported rows have `imported_from` set and
  `author_id` NULL from birth).
- Idempotent re-runs via `imported_from` source ids; progress and outcome in
  `brd_import_jobs`. No zip support (same platform limitation as Gazette -
  admins extract first).

### 5.13 URL structure

Fixed prefix from `publicBasePath: "boards"`:

| URL | Content |
|---|---|
| `/boards` | index: boards + sub-boards, search box |
| `/boards?q=…` | search results |
| `/boards/<board-slug>` | board: sub-board list + thread listing |
| `/boards/<board-slug>/<sub-slug>` | sub-board thread listing |
| `/boards/t/<thread-slug>` | thread (paginated posts, `#post-<id>` anchors) |
| `/boards/u/<username>` | public member profile |
| `/boards/feed.xml` | global RSS |
| `/boards/<board-slug>/feed.xml` | per-board RSS (public, non-noindex boards only) |
| `/boards/<board-slug>/<sub-slug>/feed.xml` | per-sub-board RSS |

`t` and `u` are reserved segments, so board slugs `t` and `u` are rejected at
creation. Thread slugs are globally unique, which keeps thread URLs stable
when a thread is moved between boards.

### 5.14 Bookmarks and read state

Members can bookmark any thread (toggle on the thread page); the bookmark
list lives on their Boards profile settings alongside subscriptions, newest
first, with unsubscribe-style removal. Bookmarks are private to the user.

Read tracking: viewing a thread page upserts the member's `brd_read_state`
row (`last_read_post_at` = the newest post timestamp visible on that page
load, so a thread with replies since the visit correctly shows unread
again). Board and sub-board listings show a per-thread unread indicator and
a per-board unread count for logged-in members, derived from the
`brd_read_state` join described in section 3. A "mark board read" action
bulk-upserts read state for the board's threads. Anonymous visitors get no
read tracking - it is session-keyed, not cookie-keyed.

## 7. Puck integration

Two distinct surfaces, following Gazette exactly:

**Thread-opener editor** - the thread composer embeds a module-internal Puck
config (pattern: `modules/gazette/components/puck/body/bodyEditorConfig.tsx`
and `bodyRscConfig.tsx`; host pattern: Gazette `PostEditor.tsx`). Editor and
RSC configs must produce identical markup and classes (core invariant). Six
blocks, none manifest-registered:

| Block | Notes |
|---|---|
| `BoardsProse` | richtext via Puck's built-in richtext field, same restricted schema as `GazetteProse` (headings 2-4, no image node) |
| `BoardsPullQuote` | mirrors `GazettePullQuote` |
| `BoardsCode` | editor renders plain `<pre>`; RSC highlights with core `shiki` (`codeToHtml` + `createCssVariablesTheme`), CSS variables namespaced `--brd-shiki-*`; pattern: `modules/gazette/components/puck/body/GazetteCode.tsx` |
| `BoardsImage` | media-picker field, mirrors `GazetteImage` |
| `BoardsPoll` | poll definition (question, options, multiple/close date); on save the thread API materialises it into `brd_polls`/`brd_poll_options`; RSC renders live results from the tables |
| `BoardsEmbed` | URL field; whitelist-rendered embeds (YouTube, Vimeo) as responsive iframes, anything else as a plain link card. No oEmbed fetching at render time |

Opener data stored as Puck Data JSON in `brd_threads.opener_data`.

**Reply composer** - replies do not use Puck. They use the same richtext
field component the Prose block wraps, standalone, storing document JSON in
`body_source` and sanitised HTML in `body_html`. See section 11 for the
editor dependency question.

**Templates** - `brd_thread_templates` stores opener Puck Data; the new-thread
screen offers a template chooser, mirroring Gazette's `posts/new`.

| Key | Description | Default expectation |
|---|---|---|
| `boards.access` | See the Boards admin nav entry and module admin screens | granted to roles that get module access; core Admin implicit |
| `boards.manage` | Structure, settings, tags, templates, imports | Admin-tier roles |

Moderator and Global Moderator are **not** core permission keys - they are
`brd_moderator_assignments` rows (section 4), assigned by core admins only.
Core admins (`isAdmin`) pass every module check unconditionally, mirroring
Gazette.

## 11. Protected Category Items - Separate Review Required

Nothing below may be implemented without Chris's explicit sign-off.

1. **Core per-user notifications (bell rework).** Core already has a
   `Notification` model (`prisma/schema.prisma`), an admin bell
   (`components/admin/NotificationBell.tsx`), `/cactus-admin/notifications`,
   and `lib/notifications/alerts.ts` - but it is admin-only (gated
   `config.manage`) and has no `userId` column. Boards needs per-user
   notifications. The extension is specified separately in
   **`NOTIFICATIONS_SPEC.md`** (DRAFT, review required): per-user rows,
   member-visible bell, read/unread, shared write helper in
   `lib/notifications/`, email hook, preferences, digest support. Boards
   consumes it through `modules/boards/lib/notify.ts` (5.7) and works
   degraded (email-only/no-op) until it ships. Touches core Prisma schema
   (init migration edit) - protected.
2. **Editor dependency (Tiptap).** Modules cannot carry npm dependencies;
   anything new lands in core `package.json` (shiki precedent). The brief
   selects Tiptap for the reply composer and richtext blocks. Mitigating
   fact found during grounding: Puck's built-in richtext field already uses
   Tiptap internally (see Gazette's Known Limitations note about
   `@tiptap/extension-image` not being a transitive dependency of
   `@puckeditor/core`), so the Gazette-style path needs **no new
   dependency**. Adding first-class `@tiptap/*` packages to core (for the
   standalone reply composer and/or an image node) is a new-dependency
   decision requiring sign-off. Recommendation: start with the
   zero-new-dependency Puck richtext path; escalate only if the reply
   composer genuinely needs more.
3. **Confirmation of non-changes.** This module requires **no** changes to
   `proxy.ts`, auth/session code, core roles/permissions models, or the core
   Prisma init migration (item 1 excepted, and it is deferred). Two
   intentional in-scope core changes are documented in the main body - the
   dashboard extension point (5.10) and the generic nested feed.xml
   delegates (5.8) - both module-name-free and touching none of the
   protected categories.
